# Project-Aware Game Task Constructor

Stage 45 связывает пользовательское намерение с фактической структурой Unreal-проекта. Конструктор не разрешает модели угадывать цель по похожему имени: игровая операция должна получить точные `node_id` и, когда он доступен, `object_path` из актуальной Project Map.

## Рабочий цикл

`Project Map -> Сфера -> Направление -> Операция -> Точный объект -> Диагностика -> Предложение -> Подтверждение -> Выполнение`

Все точки входа используют `assets/generated/leetcode/game-task-builder/state.json`:

- главная кнопка `Начать работу с проектом` на домашнем экране Unreal-проекта;
- `+ -> Конструктор задачи` возле prompt;
- постоянная вкладка `Конструктор игровой задачи` рядом с `Agent` в центре рабочего пространства;
- `Проект -> Конструктор игровой задачи`.

Сессия, proposal, собственные варианты, relation proposals и manifests восстанавливаются после перезапуска.

## Встроенная настройка Unreal

Обычному пользователю не нужны терминал, PowerShell и переменные окружения. Если проект ещё не выбран, конструктор показывает кнопку `Выбрать Unreal-проект` и список недавних рабочих папок. После выбора Leetcode автоматически:

1. находит `.uproject` в корне выбранной папки;
2. сопоставляет `EngineAssociation` с установленной Unreal Engine 5.8;
3. проверяет Editor, UnrealEditor-Cmd, UBT, UAT, Visual Studio C++, MSVC, .NET и Windows SDK;
4. создаёт безопасный локальный профиль Unreal MCP;
5. проверяет Python Editor Script, Editor Scripting Utilities и опциональный Unreal MCP;
6. предлагает одну основную кнопку `Анализировать проект`.

При первом выборе Unreal-проекта без Project Map мастер открывается автоматически. На домашнем экране всегда видна последовательность `Проект открыт -> Project Map -> Задача агенту`. До анализа основная кнопка называется `Начать работу с проектом`, а после готовности карты — `Создать задачу для игры`. Отдельная кнопка `Открыть карту проекта` становится доступной, когда найдены первые узлы.

Unreal-onboarding включается только при обнаружении настоящего `.uproject` в корне выбранной рабочей папки. Наличие исходного кода, ассетов или standalone `.uplugin` само по себе не запускает игровой Unreal-pipeline. Rust, Node/React, Python, Unity, Godot и обычные папки продолжают использовать общий агентный старт; определённый тип показывается рядом с названием проекта.

Если обязательные project-плагины выключены, действие `Настроить автоматически` сначала показывает точный список изменений. После подтверждения Leetcode создаёт копию исходного `.uproject` в `assets/generated/leetcode/backups/unreal`, меняет только allowlist плагинов и запускает Project Map scan. Произвольный plugin ID через мастер записать нельзя. MCP можно оставить выключенным; тогда доступны Project Map и базовый анализ, но не интерактивные MCP-операции редактора.

`LEETCODE_UE_VALIDATION_WORKSPACE`, `UE_ENGINE_ROOT` и скрипты из `scripts/` остаются developer/CI-интерфейсом для воспроизводимых live-тестов. Они не являются частью пользовательского рабочего процесса.

## Project Map readiness

Состояния карты: `uninitialized`, `scanning`, `ready`, `degraded`, `stale`, `failed`. Первый изменяющий Unreal-сценарий требует глубокого scan. Конструктор выполняет быстрый scanner, обнаруживает Engine/MCP, запускает `UnrealEditor-Cmd` в headless/NullRHI режиме, экспортирует Asset Registry и обновляет семантические связи. Дальнейшие refresh используют инкрементальное слияние; связи с источником `ui:*` сохраняются.

Health-report начинается с понятного вывода для разработчика: что Leetcode уже знает, чего не хватает, какие действия доступны сейчас и что нужно сделать для полной карты. Показатель готовности не является процентом просканированных файлов или готовности игры. Он складывается из четырёх проверяемых слоёв: структура проекта — 40 пунктов, Unreal Asset Registry — 35, игровые семантические связи — 15, проверка внешних ссылок — 10. Завершённая проверка ссылок даёт свои 10 пунктов даже при найденных missing targets: это диагностический результат проекта, а не отсутствие данных у анализатора.

Отчёт отдельно показывает количество узлов кода, Unreal-ассетов, planning-объектов и реально связанных узлов. `unresolved` учитывает только отсутствующие цели внутри `/Game`; зависимости Unreal Engine и подключённых плагинов показываются отдельно и не снижают готовность карты. Это не число повреждённых файлов. `ambiguous labels` означает повторяющиеся отображаемые имена, а не ошибку: точное выполнение использует `node_id` и `object_path`.

Результат анализа не привязан к открытому мастеру и не исчезает после его закрытия. При запуске и завершении scan Leetcode автоматически открывает правую вкладку `Проект`. Она показывает итоговый статус, готовность данных, состав карты, полноту каждого источника, проблемы, последний запуск и рекомендуемые следующие действия. Последний отчёт сохраняется вместе с состоянием конструктора и восстанавливается после перезапуска приложения. В чат также добавляется короткое системное сообщение о завершении анализа и месте, где доступен полный отчёт.

Статус изменения карты основан на сохранённом снимке реальных входов Unreal-проекта, а не на служебных файлах Leetcode. В снимок входят `.uproject`, `.uplugin`, `Source`, `Content`, `Config`, `Build` и используемый Asset Registry. Журналы, UI-state, Project Map JSON и другие файлы в `assets/generated/leetcode` не делают карту устаревшей. Если после анализа меняется проект, вкладка `Проект` показывает diff в стиле source control: количество и пути добавленных, изменённых и удалённых файлов. Верхняя кнопка `Обновить` только проверяет изменения относительно baseline, а действие `Синхронизировать изменения` перестраивает карту и фиксирует новый baseline.

Основной путь к `100/100` — одна кнопка `Завершить анализ автоматически`. Она запускает Unreal headless scan, получает Asset Registry, перестраивает семантические связи и повторно проверяет карту. Ручной импорт JSON и открытие Unreal Editor скрыты как запасные способы диагностики. Автоматический scan считается успешным только после появления нового `asset_registry.json`; одного нулевого exit code Unreal недостаточно. Windows verbatim-пути `\\?\...` нормализуются перед передачей в `UnrealEditor-Cmd`, чтобы Python exporter не разрешался относительно каталога Engine. Если Engine или MCP недоступны, пользователь может настроить их либо продолжить анализ без разрешения изменяющих игровых операций.

Большие реальные Asset Registry exports поддерживаются до 96 МБ. Пустой `object_path` из UE 5.8 восстанавливается из `package_name` и `asset_name`. Новый exporter индексирует проектный mount `/Game`, получает нужные семантические tags через `AssetData.get_tag_value` и не раздувает карту всем содержимым Engine.

Визуальная карта открывается в режиме `Обзор`: восемь смысловых групп вместо тысяч узлов на одном полотне. Клик по группе открывает `Детали`, где применяются поиск и фильтр. Граф кэшируется в памяти, связи отбираются только для видимых узлов, подписи скрываются на дальнем масштабе, а колесо мыши масштабирует относительно курсора. JSON больше не читается и не пересканируется в каждом UI-кадре.

Статусы `degraded` и `failed` не оставляют пользователя на пустом экране, если быстрый scanner уже нашёл узлы. Каталог сфер и операций остаётся доступным для изучения, UI показывает причину и длительность последнего анализа, а выполнение изменяющей задачи по-прежнему требует полной актуальной карты. Readiness и статический каталог кэшируются в состоянии интерфейса: сканирование Unreal запускается только явным действием или обновлением проекта, а не повторяется в каждом кадре UI.

## Каталог и контракты целей

Каталог содержит 8 сфер, по 10 направлений и 10 стандартных операций: 800 конечных сценариев. Каждая операция хранит допустимые типы узлов, потребность в Skeleton, поддержку группы целей, артефакты, инструменты, validation, capability и risk class.

Target Resolver читает только текущую Project Map. Несколько персонажей дают `ambiguous` и требуют явного выбора одного персонажа или группы. `UnrealStaticMesh` детерминированно исключён из целей персонажной анимации.

`Свой вариант` действует в текущей задаче. После подтверждённого выполнения UI отдельно предлагает сохранить его в каталог проекта; повторное сохранение того же варианта не создаёт дубликат.

## Диагностика и proposal

Проверка возвращает `ready`, `needs_setup`, `ambiguous`, `stale_context` или `external_tool_required`. Проблема не блокирует пользователя без объяснения: каждый вариант подготовки содержит действия, сложность, риск, инструменты, ожидаемое время и необходимость approvals.

Для отсутствующего Skeleton доступны создание, выбор совместимого, импорт ригованной модели, временный Mannequin, смена цели/операции и собственный вариант. Выбранная remediation автоматически входит в подготовительные шаги proposal.

Карточка подтверждения показывает понимание задачи, точные и исключённые цели, зависимости, этапы, подготовку, артефакты, validation, риски, улучшения и варианты ускорения. Улучшения и субагенты по умолчанию выключены.

## TaskManifest и безопасность

`Подтверждаю` создаёт immutable `TaskManifest` с:

- ID сессии и операции;
- точными и разрешёнными node IDs;
- Unreal object paths;
- выбранными улучшениями и способами ускорения;
- fingerprint Project Map на момент подтверждения.

Перед изменяющим файловым, shell, Unreal, desktop, asset или MCP-действием dispatcher сверяет fingerprint и явно упомянутые пути с manifest. Изменение карты или обращение к постороннему узлу требует нового proposal и повторного подтверждения. После завершения запуска manifest остаётся в истории, но снимается с активного исполнения.

MCP получает два независимых protocol metadata блока: `com.leetcode/projectNodeContext` и `com.leetcode/taskManifest`. Аргументы удалённого инструмента не модифицируются.

- `game_task_catalog_snapshot`
- `resolve_game_task_targets`
- `evaluate_game_task_prerequisites`
- `prepare_game_task_proposal`
- `propose_project_relation`

Инструменты имеют action-specific JSON schemas, governance category `game_task_builder` и записи `game_task_builder\t...` в audit journal.

## Unreal Engine 5.8 live checklist

1. В Leetcode выберите отдельный тестовый UE 5.8 проект через встроенный мастер.
2. Если Python Editor Script Plugin или Editor Scripting Utilities выключены, подтвердите автоматическую настройку и проверьте созданную резервную копию `.uproject`.
3. Нажмите `Анализировать проект` без запуска терминала.
4. Проверьте состояние `ready`, registry path, Engine, MCP, число узлов и semantic edges.
5. Убедитесь, что `BP_Hero`, `SK_Hero`, `SKEL_Hero` и `ABP_Hero` связаны через dependency, `uses_skeleton`, `compatible_with` и `animates`.
6. Выберите операцию локомоции. `BP_Hero`/`SK_Hero` должны быть кандидатами, `SM_Bucket` должен отображаться только как исключённый Static Mesh.
7. Удалите или отвяжите Skeleton в копии fixture и проверьте все guided remediation options.
8. Подготовьте proposal, не выбирайте улучшения и подтвердите. В manifest не должно быть improvement IDs.
9. Попробуйте MCP-вызов к `SM_Bucket`: dispatcher должен отклонить посторонний object path.
10. Измените Project Map после подтверждения: следующий mutating tool должен потребовать обновление привязки.
11. Перезапустите Leetcode до подтверждения и проверьте восстановление сессии конструктора.

Live scan запускает локальный Unreal-процесс и должен выполняться осознанно на копии проекта. Rust unit-тесты и fixture не заменяют визуальную проверку результата в Editor.
